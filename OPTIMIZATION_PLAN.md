## 三、技术方案（四项优化）

### 1. CFD/FEM温度求解作为验证标准 [P1]

#### 问题
当前引擎使用半经验模型。关键缺陷：
- 未求解柱坐标下的完整热传导方程（缺少 `(1/r)∂T/∂r` 径向项）
- 边界条件用经验系数模拟，非精确 Dirichlet/Neumann
- 无验证手段衡量模拟精度

#### 方案

**后端严格FDM求解器（Python）**

```
文件: server/solver.py
依赖: numpy, scipy, fastapi, uvicorn
```

核心方程（柱坐标2D轴对称）：
```
∂T/∂t = α · (∂²T/∂r² + (1/r)·∂T/∂r + ∂²T/∂h²)
```

边界条件：
- 壁面：`-k·∂T/∂r|_wall = h_wall · (T_wall - T_ambient)`（第三类边界）
- 液面：`-k·∂T/∂h|_surface = h_surf · (T_surface - T_ambient)`（蒸发散热）
- 中心轴：`∂T/∂r|_r=0 = 0`（轴对称）

**前端UI集成**
- 右上角新增精度指标徽章：`模拟精度: ±X°C (vs CFD)`
- 每次求解完成后自动/手动发起验证请求
- 偏差曲线叠加在温度-时间图上（虚线表示CFD参考值）

**技术决策：为什么不直接用FEM？**
FDM在结构网格上精度足够且速度快，与当前数学形式一致。FEM留给更复杂的非规则几何（目标2）。

### 2. 图纸上传→网格→3D展示→空间温度场 [P2]

#### 分阶段实施

**Phase 1：浏览器端2D截面绘制（纯前端，可独立交付）**

```
新增文件: mesh-editor.js
```

工作流：
1. 用户通过 Canvas 绘制截面轮廓（鼠标绘制 或 拖入参考图描边）
2. 提取轮廓点集 → 旋转生成轴对称3D几何
3. 自动生成结构化网格（20×24或自适应密度）
4. 在自定义几何上运行FDM求解
5. Three.js 渲染几何 + 温度彩色映射

核心交互：
```
┌──────────────────────┐
│  绘图模式 / 导入模式   │
│  ┌──────────────────┐│
│  │   Canvas 绘图区    ││
│  │   左键: 加点       ││
│  │   右键: 闭合       ││
│  │   拖拽: 调整轮廓    ││
│  └──────────────────┘│
│  [重置] [撤销] [生成] │
└──────────────────────┘
```

**Phase 2：STL导入+后端网格划分（需后端）**
- 后端用 meshio + GMSH API 自动划分非结构网格
- scikit-fem 或 FEniCS 求解
- 结果回传前端

**交互限制说明**
| 功能 | 可行性 | 实现方式 |
|------|--------|----------|
| 2D截面绘制→轴对称3D | ✅ 纯前端 | Canvas + Three.js LatheGeometry |
| STL解析渲染 | ✅ Three.js STLLoader | 已有成熟库 |
| 非结构网格划分 | ❌ 前端不可行 | 后端 GMSH API |
| 3D FEM求解 | ❌ 前端不可行 | 后端 scikit-fem |

### 3. 质感优化 [P1]

#### 3.1 玻璃拟态面板
```css
.glass-panel {
    background: rgba(255,255,255,0.72);
    backdrop-filter: blur(20px);
    -webkit-backdrop-filter: blur(20px);
    border: 1px solid rgba(255,255,255,0.3);
    box-shadow: 0 8px 32px rgba(0,0,0,0.08);
}
```
应用范围：左侧参数面板、右侧图表面板、状态栏

#### 3.2 3D场景加强
| 改进项 | 实现方式 |
|--------|----------|
| HDR环境贴图 | `PMREMGenerator` + EXR环境图 |
| 液面反射 | `MeshPhysicalMaterial` 设置 `transmission`, `roughness`, `metalness` |
| 柔和阴影 | `renderer.shadowMap` + 聚光灯阴影贴图 |
| 地面反射 | 半透明反射平面 |

#### 3.3 颜色映射升级
当前：分段线性映射（红→橙→黄→绿→蓝）
替换：perceptually uniform colormap（Inferno / Viridis）

```javascript
// 引入 color-map.js 依赖
const cmap = getColormap('inferno', 256);
const color = cmap[normalizedTemp * 255];
```

Inferno 在工业/科学模拟中视觉专业度最高，且对色觉障碍友好。

#### 3.4 微交互
- 滑块追踪值显示 + 光晕跟随
- 按钮点击 ripple 效果
- 探针放置时的飞线动画（从探针面板到3D位置）
- 参数变化时 liquid mesh 做 morph 过渡（非立即重建）
- 时间轴拖拽时温度场实时渐变

#### 3.5 粒子系统（蒸发效果）
使用 Three.js Points，在液面位置随机生成上升粒子：
- 粒子大小随高度衰减
- 透明度随高度渐变消失
- 粒子密度与 `T_liquid - T_ambient` 正相关

## 六、关键技术决策

| 决策 | 选项 | 结论 | 理由 |
|------|------|------|------|
| 后端框架 | Flask vs FastAPI | **FastAPI** | 异步+自动文档，适合科学计算API |
| PINN框架 | PyTorch vs TensorFlow | **PyTorch** | 研究生态更广，调试方便 |
| 颜色映射 | Inferno vs Viridis | **Inferno** | 工业视觉更专业，低亮度区对比度更好 |
| 3D渲染加强 | 纯Three.js增改 vs 换库 | **增改Three.js** | 已有完整实现，换库成本高 |
| 网格划分 | 前端做 vs 后端做 | **前端(简单)/后端(复杂)** | 分阶段策略，前端做轴对称足够 |
| 状态管理 | 全局对象 vs 框架 | **全局对象** | 项目规模小，引入框架增加复杂度 |
